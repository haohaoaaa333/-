# 考公宝 V2.0 云函数重构设计

> 配套文档：《考公宝 V2.0 数据库设计说明书》《考公宝 V2.0 题库导入系统重构方案》
> 目标：在不推翻微信云开发的前提下，按 V2 数据库设计重构云函数，打通
> `PDF → import_tasks → MinerU → draft_assets → question_drafts → AI → 审核 → questions → 刷题 → AI学习分析` 闭环。
>
> 本文档是**现状盘点 + 重构方案**，落地代码前先对齐命名与边界。

---

## 0. 核心结论（先看这段）

代码里**已经实现了 V2 草案的核心骨架**（import_tasks / workerGateway / drafts-v2），但存在三处必须重构的冲突：

1. **导入落库命名未统一**（P0-A 已解决）：V2 草稿发布与"手动上传行测包"共用 `xingce.js` 落库引擎写 `questions`，其目标集合已统一为 V2 命名（`papers`/`paper_groups`/`question_solutions`/`question_media`/`import_jobs`）；旧 `importQuestions` 直灌 `questions` 已在 P0-A2 改为先写 `question_drafts`。
2. ~~**答题集合重复**~~ ✅ 已收敛（P1）：单题级 → `practice_records`，会话级汇总 → 独立 `practice_sessions`，`user_answers` 已废弃删除。
3. **设计要求的 3 个缺口集合（P0-B 已完成）**：`draft_assets`、`wrong_questions`、`knowledge_points` 均已建并接入写入链路。

重构不是"从零写"，而是**统一命名、收敛重复、补齐缺口、固化接口契约**。

### 1.3 AI 结构化实际运行位置

**不在云端函数内**，而在本机流水线：

```
PDF → mineru_worker.py(MinerU识别) → structure_questions.py(AI结构化)
     → 写 question_drafts(+ draft_assets)
     → 调用 workerGateway.complete 回写 import_tasks
```

`scripts/structure_questions.py` 是关键环节，由 `admin/local-server.js` 编排。因此"AI结构化接口"的重构重点是**固化本地流水线与云端的契约**（入参/出参/落库集合），而非新开一个云函数。

## 4. 关键接口契约

### 4.1 MinerU 任务接口（已就绪，仅列要点）

`workerGateway` 已实现：HMAC 签名鉴权 + nonce 防重放 + 租约(lease_token/lease_expires_at) + 状态机。
- `claim`：事务领取 `waiting` 任务，写 `claimed` + 租约
- `heartbeat`：上报 `mineru_processing`/`splitting` 进度与 `processed_pages`
- `complete`：任务置 `draft_ready`，回写 `draft_paper_id`/`question_count`/`asset_bundle_id`
- `fail`：置 `failed`，记录 `error{stage,code,message,retryable}`

### 4.2 AI 结构化接口（本地为主，云端为辅）

本地 `structure_questions.py` 必须：
- 读 MinerU 产物 → 产出 `question_drafts`（一题一档，`schema_version:"2.0"`，`answer.verified` 三态：`verified`/`unverified`/`missing`）
- 同步写 `draft_assets`（证据链：pdf_page / pdf_image / mineru_markdown / layout_json / bbox）
- 通过 `admin.draft.create` + `admin.draft.append`（分批，绕过 100KB HTTP 上限）落库
- 完成后调 `workerGateway.complete`

可选 `aiStruct` 云函数：重跑单题/整卷 AI、补 `ai{model,prompt_version,confidence,request_id}` 字段，便于换模型/查成本/重跑。

### 4.3 审核接口

`admin.draft.*` + `review_events`：
- `draft.get` / `draft.listQuestionDrafts`：取待审题目
- `draft.review`（含 `reviewWithGemini` AI 预审）：修改即写 `review_events{question_id,operator,action,before,after}`
- `draft.setStatus`：pending→approved/failed
- 发布闸门：`validateForApproval` 已强制"题干非空 / A-D 四选项完整 / 答案已确认 / 解析非空"，继续沿用

### 4.4 发布接口

`admin.draft.publishDraft` → 统一落库目标：
- 写 `papers`（发布卷元数据：name/category/year/total_questions/source/status）
- 写 `questions`（**统一 V2 schema**，见下）
- 写 `knowledge_points`（题目的知识点去重归集）
- 置 `import_tasks` 为 `completed`，`draft_papers` 为 `published`

> 旧 `xingce.js` 的 `importXingcePackage` 在迁移完成后删除。

### 4.5 权限控制（三层）

| 层 | 机制 | 状态 |
|----|------|------|
| Worker → 云端 | HMAC `WORKER_SECRET` + 时间戳(±5min) + nonce 一次性 | ✅ workerGateway 已做 |
| 管理员 → 后台 | `admin_users` 白名单（`admin/lib/auth.js`） | ⚠️ 须确认覆盖所有 `admin` action |
| 用户 → 自身数据 | `wxContext.OPENID` + `_openid` 过滤（practice_records / practice_sessions / wrong_questions / favorites / user） | ✅ 全部已强制 `_openid` 过滤 |
| 数据库安全规则(acl) | 集合级 read/write 规则 | ❌ `cloudbaserc.json` 未配置，需补（questions 公开；其余限 _openid/admin） |

> `questions`/`papers`/`knowledge_points` 等公开只读（acl `READONLY`）；`users`/`practice_records`/`practice_sessions`/`wrong_questions`/`favorites` 用户侧限本人（acl `PRIVATE`）；`draft_*`/`import_tasks`/`review_events`/`admin_users` 限 admin 服务端（acl `PRIVATE`）。完整 acl 见 `cloudbaserc.json` 与《V2.0-数据库索引与权限清单.md》。

### 4.6 学习分析接口（P2）

`practice.analyze`：数据流末端的学习分析，**确定性聚合**（不调用大模型，省 AI 成本；AI 结构化本身走本机流水线，决策 #3）。

**入参**：`{ openid(由 wxContext 注入), module_id?(可选, 按模块过滤) }`

**数据来源**：
- `practice_records`（单题级，`_openid` 过滤）→ 提取 `question_id` / `is_correct` / `module_id`
- `questions`（按 `question_id` in 批量取）→ `knowledge_points[]` / `type` / `difficulty` / `module_id` / `content`
- `wrong_questions`（`_openid` 过滤）→ 取「未掌握且 `wrong_count>=2`」的反复错题

**聚合口径**：
- `module_stats` / `type_stats` / `difficulty_stats`：各维度 `attempts` / `correct` / `accuracy=round(correct/attempts*100)`
- `weak_knowledge_points`：每个知识点 `attempts` / `correct` / `accuracy` / `weakness=round((1-accuracy)*100)`，按 `weakness` 降序（同分按 `attempts` 降序），取前 15，附 `sample_question_ids`（前 5）
- `repeated_wrong`：反复错题，`wrong_count` 降序，取前 20，附 `content_preview`（题干前 60 字）
- `review_plan`：`knowledge_points` = `weakness>=30` 的薄弱点取前 8；`question_ids` = 反复错题 + 薄弱点样本题去重截断 30

**返回**：`{ total_practiced, overall_accuracy, module_stats[], type_stats[], difficulty_stats[], weak_knowledge_points[], repeated_wrong[], review_plan{ knowledge_points[], question_ids[] } }`；无记录返回 `emptyAnalysis`（`code:0, message:'no_records'`）。

**性能**：`practice_records` 与 `wrong_questions` 均分页 `fetchAll`（批量 100 + `_id` 去重，上限 5000）；`questions` 按 `_id in` 批量取（批量 100，仅投影分析所需字段）。无新增集合（按需计算）；如需降低高频调用开销，后续可加 `learning_reports` 缓存集合（按用户存最近一次分析结果 + `computed_at`），属可选增强。

**前端**：`src/api/practice.ts` 新增 `getPracticeAnalysis(moduleId?)` → `callCloudFunction('practice', { action:'analyze', ... })`；`practice` 已在 `base.ts` 个人数据函数白名单内，隐私未开启时自动跳过。

## 5. Schema 统一（消除漂移）

### 5.1 `questions`（发布表）统一为 V2

```json
{
  "_id": "q001",
  "paper_id": "paper001",
  "question_no": 66,
  "module_id": "mod_language",
  "type": "single_choice",
  "stem_blocks": [{ "type": "text", "content": "题干" }],
  "options": [{ "key": "A", "content": "选项A" }, ...],
  "answer": 0,
  "answer_letter": "A",
  "analysis": "解析",
  "knowledge_points": ["工程问题"],
  "difficulty": "medium",
  "status": 1,
  "content": "题干(兼容)",
  "material": "",
  "options_text": ["选项A", ...],
  "explanation": "解析(兼容)",
  "answer_index": 0,
  "year": 2025,
  "paper_name": "2025国考"
}
```

> **P0-C 已落地**：`questions` 发布时由 `xingce.importXingcePackage` 投影为上方结构——V2 规范字段（`stem_blocks`/`options:[{key,content}]`/`analysis`/`knowledge_points`/`paper_id`/`question_no`）为权威，legacy 兼容字段（`content`/`material`/`options_text`/`explanation`/`answer_index`）保留以便过渡。
> **`answer` 决策**：设计写 `"A"`，但实际 `answer` 保留 **0-3 索引**（`answer_letter` 提供字母），原因见 §8 路线图 P0-C 决策——避免打破 `practice.submitAnswer` 的索引判分闭环。前台经 `questions/formatQuestion` 读取，输出形状不变，小程序 `src/` 无需改动。

### 5.2 `users` 对齐 V2

```json
{
  "vip": { "level": 0, "expire_time": null },
  "study": { "total_questions": 0, "correct_count": 0, "days": 0 },
  "status": 1
}
```

> 当前 `vipStatus`/`today_done` 等字段，迁移脚本双写后切换。

### 5.3 AI 字段预留（所有 AI 产出）

`ai: { model, prompt_version, confidence, request_id }` 挂在 `question_drafts` 与各 AI 产出文档，便于重跑/审计。

## 8. 分阶段路线图

- **P0-A 收敛命名** ✅：`xingce.js` 落库集合重命名为 V2 命名（`xingce_papers`→`papers` 等）；保留 `xingce.js` 作为落库引擎（内部调用 `importXingcePackage`）。
- **P0-A2 废弃直灌** ✅：`importQuestions` 不再直写 `questions`，改为按 `itemId(draftId, questionId)` 写 `draft_papers`+`question_drafts`（与 `drafts-v2` 同 `_id` 算法、`review_status:"pending"`、`answer_verified` 由原始选项字母决定），由后台草稿审核→publish 落正式库。✅ 已落地 `cloudfunctions/importQuestions/index.js`，并放宽 `validatePackage` 对资料分析组「恰好 5 题」为「至少 1 题」（真实批次按 material 分组为 6~60 题）。
- **P0-B 补齐缺口集合** ✅ 已完成：建 `draft_assets` / `wrong_questions` / `knowledge_points` 并接入写入（`importQuestions` 写证据链、`practice` 写错题本+标记掌握、`xingce.importXingcePackage` 发布时归集知识点）。
- **P0-C 统一 questions schema** ✅ 已完成：`xingce.importXingcePackage` 发布时把 `questions` 投影为 V2 规范（`paper_id`/`question_no`/`stem_blocks`/`options:[{key,content}]`/`answer`+`answer_letter`/`analysis`/`knowledge_points`/`status:1`），保留 legacy 兼容字段；`questions/index.js` 的 `formatQuestion` 改为「V2 优先、legacy 回退」双感知，前台输出形状不变；4 处查询 `.field` 投影补 `stem_blocks/analysis/knowledge_points`，`getByModule` 裸 `q.options` 改为 `projectOptions(q)` 映射。
  - **`answer` 字段决策（重要）**：设计写 `answer:"A"`，但 `practice.submitAnswer` 用 `qRes.data.answer === selected_option` 判分且前台 `selected_option` 为 0-3 索引。为避免打破线上刷题闭环，**`answer` 保留 0-3 索引作为判分真值**，并新增 `answer_letter`（"A"…）满足 V2 字母要求，供外部 API 使用。若后续确认小程序 `selected_option` 也改为字母，再把 `answer` 切到字母（待前端改造时一并切换，不阻塞本次重构）。
