# 考公宝 V2.0 题库导入系统重构方案

> 版本：v2.0-draft-1　·　范围：题库导入链路（PDF → MinerU → 切题 → AI → 草稿 → 审核 → 发布）
> 结论先行：**架构方向正确，不推翻。核心病根是"前段数据没有向发布端既有的 V2 标准收敛"。本方案让全链路统一到一套 Schema，并补齐任务化、证据链、一题一档三块企业级能力。**
>
> 配套接口设计：[考公宝V2.0云函数重构设计.md](./考公宝V2.0云函数重构设计.md)

---

## 0. 现状判断（一句话）

发布端（`cloudfunctions/admin/features/xingce.js` 的 `validatePackage` / `importXingcePackage`）**早已定义并强制了一整套 V2 标准**，但前段 MinerU → OCR → `ai_result` 完全没往这套标准上靠，中间缺一层"确定性转换 + 结构对齐"，导致两个子系统看似连着、实则断开。

**这不是架构错，是第一版跑通流程后缺少企业级数据规范的正常阶段。**

### 0.1 两套数据现状对照

| 维度 | 现状（前段 OCR 产物） | 发布端已强制要求（目标） |
|---|---|---|
| 顶层结构 | 顶层直接 `questions[]`，无 `package` 包裹 | `{schema_version:2, paper, groups[], questions[], solutions[], media[]}` |
| 版本标记 | 无 `schema_version` | `Number(schema_version)===2`，否则 422 |
| 试卷对象 | 无 `paper` | `paper._id` + `paper.title` 必填 |
| 单题结构 | `{_id, ai_result:{stem, options:[{key,text}], answer, module, images, material, type}}` | `{_id, content \| stem_blocks[], options_v2[4], answer:0-3, answer_verified, module_id, group_id}` |
| 选项 | `ai_result.options`（`{key,text}`） | `options_v2[]`（`{key, text \| content_blocks[]}`），必须 4 个且非空 |
| 答案 | `ai_result.answer`，**未识别时被填成 A（危险）** | `answer:int(0-3)` + `answer_verified:bool`；`answer_verified===false` 直接报「答案缺失」 |
| 解析 | 混在题内/缺失 | 独立 `solutions[]`，数量必须 == 题数，`question_id` 关联 |
| 资料分析 | material 归属靠启发式 | `groups[module_id==='mod_data']` 必须恰好关联 5 道小题 + 材料非空 |
| 图片 | `ai_result.images` 散落 | `media[]` 按 `asset_id` 登记，块内 `{type:'image', asset_id}` 引用，缺失即报错 |
| 证据链 | 无 | 无（本方案新增） |

> 关键洞察：**"答案禁止自动填 A" 这条，发布端本就有 `answer_verified` 支持**——是前段没有把"未识别"如实标出来，反而伪造了答案。对齐即可，无需另造字段。

## 1. 目标架构（重构后链路）

```
PDF
 │
 ▼
import_tasks           ← 任务化：一次导入 = 一条任务，可重试/取消/去重
 │
 ▼
MinerU 原始数据         ← draft_assets：markdown / layout.json / bbox / 页图 / MinerU 版本
 │
 ▼
确定性切题（Python）     ← split_questions.py，只做"按规则切"，不猜答案
 │
 ▼
Draft V2（统一 Schema） ← 一题一档 question_drafts，answer 缺失就是 null
 │
 ▼
AI 审核（异步任务）      ← ai_review_tasks：pending→processing→success/failed，不阻塞浏览器
 │
 ▼
人工三栏审核            ← 左PDF原图 / 中OCR文本 / 右结构化编辑器
 │
 ▼
question_release        ← 发布闸口：组包 → previewXingcePackage 预检 → importXingcePackage
 │
 ▼
questions 正式题库
```

**设计红线**：AI/OCR 结果**绝不直进正式库**，必须过"草稿 → 人工审核 → 发布"闸口（这一条现有 `drafts.js` 已正确坚持，继续保留）。

## 2. P0 · 统一题目 Schema（Draft V2）

### 2.1 单题文档（`question_drafts` 集合，一题一档）

```jsonc
{
  "_id": "qd_<paperId>_066",          // 全局唯一，含来源试卷与题号
  "schema_version": "2.0",

  // —— 归属 ——
  "task_id": "imp_1784xxxx",          // 属于哪次导入任务
  "paper_id": "paper_2023_xxx",       // 属于哪套试卷（draft_papers._id）
  "group_id": null,                   // 资料分析等共享材料题组；非组题为 null

  // —— 题目本体 ——
  "question_no": 66,
  "type": "single_choice",            // 枚举，见 2.4
  "module_id": "mod_data",            // 模块，见 2.3；未判定时为 null

  "stem_blocks": [                    // 有序图文块，替代裸 stem 字符串
    { "type": "text", "content": "甲、乙、丙三人……" },
    { "type": "image", "asset_id": "asset_ab12", "bbox": [x,y,w,h] },
    { "type": "formula", "latex": "\\frac{a}{b}" }
  ],

  "options_v2": [                     // 恰好 4 个（单选）
    { "key": "A", "content": "……", "content_blocks": [] },
    { "key": "B", "content": "……", "content_blocks": [] },
    { "key": "C", "content": "……", "content_blocks": [] },
    { "key": "D", "content": "……", "content_blocks": [] }
  ],

  "media": [                          // 本题引用的图片（登记在 draft_assets，按 sha256 去重）
    { "type": "image", "asset_id": "asset_ab12", "sha256": "…", "role": "stem" }
  ],

  // —— 答案（核心：禁止伪造）——
  "answer": {
    "value": null,                    // 0-3 或 null；null = 未识别
    "verified": false,                // 人工/可信来源确认才 true
    "source": "missing"              // missing | ocr | ai | human
  },

  // —— 解析与元信息 ——
  "analysis": null,                   // 有序块或 null，缺失就是 null
  "analysis_blocks": [],
  "knowledge_points": [],
  "difficulty": null,                 // 1-5 或 null
  "confidence": 0.0,                  // 切题/结构化置信度 0-1

  // —— 审核状态（一题一档后从整卷 review map 下沉到题上）——
  "review_status": "pending",         // pending | approved | rejected | needs_fix
  "review_note": "",

  // —— 证据链回指 ——
  "evidence": {
    "source_page": 12,                // PDF 第几页
    "mineru_block_ids": ["b_045"],    // 对应 MinerU layout 块
    "asset_bundle_id": "da_xxx"       // draft_assets 文档
  },

  "created_at": 0,
  "updated_at": 0
}
```

> 彻底废弃 `{"ai_result":{}}` 这种"黑盒袋子"。MinerU、确定性切题、AI、人工审核、发布，**全部读写同一套字段**。

### 2.2 答案三态（P0，最重要的一条纪律）

| 场景 | 旧（危险） | 新（正确） |
|---|---|---|
| OCR 明确识别到答案 | `answer:"A"` | `{value:0, verified:false, source:"ocr"}` |
| OCR/AI 未识别 | `answer:"A" + 解析:"未识别"` ❌ | `{value:null, verified:false, source:"missing"}` |
| 人工确认 | — | `{value:0, verified:true, source:"human"}` |

**发布端硬闸**：`validatePackage` 已有 `answer_verified===false → 报错「答案缺失或不是A-D」`，且 `answer` 必须是 0-3 整数。所以组包时映射规则为：`answer.value → question.answer`（null 时置一个占位并让 `answer_verified=false` 拦下），`answer.verified → question.answer_verified`。**没有答案的题，永远发布不出去**——这正是我们要的。

### 2.3 模块映射（`module` 名 → `module_id`）

以现有模块常量为准，切题脚本统一输出 `module_id`：

| 中文模块 | module_id | 题量（法检卷基线） | 特殊约束 |
|---|---|---|---|
| 政治理论 | `mod_politics` | 20 | — |
| 常识判断 | `mod_common` | 15 | — |
| 言语理解 | `mod_language` | 30 | — |
| 数量关系 | `mod_quantity` | 10 | — |
| 判断推理 | `mod_judge` | 35 | 图形推理选项为图 → `options_v2[].content_blocks` |
| 资料分析 | `mod_data` | 20 = 4组×5题 | 每组 `group_id` 关联恰好 5 题 + 材料非空 |

> 注：`mod_data`、`mod_language`、`mod_quantity` 已在现有代码/产物中确认使用；其余 id 需与前端模块定义对齐，落地前用一处常量表统一（见 §7 改造清单）。

### 2.4 题型枚举

`single_choice`（单选，默认）｜ `data_group`（资料分析共享材料组，落在 `draft_papers.groups`）｜ 预留 `multi_choice` / `fill_blank`。

## 3. P0 · 证据链（`draft_assets`）

正式题目必须可回溯：`正式题 → 原始PDF页 → MinerU文本 → AI结果 → 人工修改记录`。

```jsonc
// draft_assets：一次导入的原始物料登记（可回放、可追责）
{
  "_id": "da_1784xxxx",
  "task_id": "imp_1784xxxx",
  "paper_id": "paper_2023_xxx",
  "pdf_sha256": "…",                 // 用于去重（见 §6）
  "mineru_version": "mineru-x.y.z",
  "assets": {
    "markdown_path": "cloud://.../full.md",
    "layout_json_path": "cloud://.../layout.json",
    "page_images": ["cloud://.../p1.png", "…"],   // 每页 PDF 截图
    "figures": [                                   // 抽取出的图片
      { "asset_id": "asset_ab12", "sha256": "…", "src": "cloud://…", "bbox": [x,y,w,h], "page": 12 }
    ]
  },
  "created_at": 0
}
```

- 图片沿用现有 `question_media` 的 **SHA-256 去重**思路，`draft_assets.figures[].asset_id` 与题内 `media[].asset_id` 一致，发布时直接映射进 `media[]`。
- 人工修改记录不塞进本表，独立进 `review_events`（§4.5），保证"物料"与"操作日志"分离。

## 4. P1 · 数据库集合重设计（一题一档）

**告别"一个 draft 文档塞整套试卷"**（现状 `question_drafts` 单 doc 存整卷 `package`，1000 页教材会撑爆单文档、审核也难）。拆成 5 张表：

```
import_tasks         一次导入任务（生命周期总控）
   │ 1─N
draft_papers         一套来源试卷的元数据（含 groups 资料组）
   │ 1─N
question_drafts      一题一档（§2.1）
   │ 1─1
draft_assets         原始物料/证据链（§3）
   │ 1─N
review_events        审核操作流水（谁、何时、改了什么）
```

### 4.1 `import_tasks`

```jsonc
{
  "_id": "imp_1784xxxx",
  "source": "pdf",
  "file_name": "2023法检真题.pdf",
  "pdf_sha256": "…",
  "status": "pending",       // 见 §5.1 状态机
  "progress": { "stage": "mineru", "percent": 40, "message": "解析第12/25页" },
  "paper_id": "paper_2023_xxx",
  "counts": { "questions": 0, "with_answer": 0, "need_review": 0 },
  "error": null,
  "retry_count": 0,
  "created_at": 0, "updated_at": 0
}
```

### 4.2 `draft_papers`

```jsonc
{
  "_id": "paper_2023_xxx",
  "task_id": "imp_1784xxxx",
  "title": "2023年法检系统行测真题",
  "module_counts": { "mod_data": 20, "mod_judge": 35, "…": 0 },
  "groups": [                       // 资料分析共享材料组（对齐发布端 xingce_question_groups）
    { "_id": "grp_01", "module_id": "mod_data",
      "material_text": "", "material_blocks": [], "question_ids": ["qd_..._066","…×5"] }
```

索引建议：`task_id`、`paper_id`、`module_id`、`review_status`、`(paper_id, question_no)` 唯一。

### 4.5 `review_events`（审核流水）

```jsonc
{
  "_id": "rev_1784xxxx",
  "question_id": "qd_..._066",
  "paper_id": "paper_2023_xxx",
```

## 5. 状态机定义

### 5.1 `import_tasks.status`

```
pending ──領取──▶ mineru ──▶ splitting ──▶ ai_review ──▶ human_review ──▶ ready ──发布──▶ published
   │                │            │             │              │
   └──cancel──▶ canceled        └──────────── failed ◀────────┘（任一阶段异常）
                                       │
                                   retry（retry_count+1）──▶ 回到失败阶段
```

### 5.2 `question_drafts.review_status`

`pending → approved | rejected | needs_fix`；只有全部非 `pending/rejected` 且各题 `answer.verified===true` 才允许该卷进入 `ready`。

### 5.3 `ai_review_tasks.status`（§8 详述）

`pending → processing → success | failed`；失败可重试，浏览器不等待，轮询/推送结果。

## 6. P1/P2 · MinerU Worker 改造（适配 RTX 5070 8G）

现状：识别偶发 CUDA OOM（25页真题易爆 8G 显存），且是"同步等待 + 无重试"。改造为任务队列 + 单 GPU 串行：

```
import_task(pending)
   ▼ Worker 轮询领取（原子占用，防并发）
GPU 锁（同一时刻仅 1 个 MinerU 作业占用显存）
   ▼
MinerU 解析（失败 → 自动降批/降分辨率 → 重试，retry_count≤N）
   ▼
上传结果到 draft_assets + 云存储
   ▼
更新 import_task.progress / status
```

必须补齐的能力：

| 能力 | 做法 |
|---|---|
| 单 GPU 并发锁 | 内存/文件锁，保证同一时刻只跑一个 MinerU 作业，避免 8G 显存被多任务挤爆 |
| 失败重试 | OOM/CUDA 错误捕获后自动重试；可选降页批（分段送入）或切 CPU 模式兜底 |
| 取消任务 | `import_task` 置 `canceled`，Worker 检查到即中断并释放 GPU 锁 |
| PDF hash 去重 | 入队前算 `pdf_sha256`，命中已成功任务直接复用 `draft_assets`，不重复烧显存 |
| 缓存自动清理 | 任务完成/失败后清理 MinerU 中间目录，保留登记在 `draft_assets` 的产物 |

## 7. P2 · AI 审核闭环（异步任务，不占浏览器）

现状：Gemini 仅"逐题按钮触发、浏览器同步等"，没有形成闭环。改造为后台任务：

```
question_draft ──提交──▶ ai_review_tasks(pending)
                            ▼ 后台 Worker 领取
                         processing（调多模态模型：读页图+OCR文本 一起判）
                            ▼
                         ai_review_results（建议答案/建议解析/存疑标记 + 置信度）
                            ▼
                         success → 人工在三栏界面确认（AI 只建议，绝不自动通过）
                            └ failed → 可重试
```

要点：AI 结果写 `ai_review_results` 并生成一条 `review_events(actor:"ai:gemini", action:"ai_suggest")`；**永不自动改 `answer.verified=true`**，确认权始终在人工。

## 10. 分阶段实施路线图

| 阶段 | 目标 | 交付 | 判定通过 |
|---|---|---|---|
| **P0-A** | 统一 Draft V2 Schema + 答案三态 | 改 `split/structure` 输出 + 组包转换层 | 一套 OCR 真题能组出通过 `previewXingcePackage` 的 V2 包（答案缺失的题被正确拦下，而非填 A） |
| **P0-B** | 证据链 `draft_assets` | 落库原始 md/layout/页图/图片+版本 | 任一正式题可回溯到 PDF 页与 MinerU 块 |
| **P1-A** | 数据库拆分（一题一档 5 表） | `import_tasks/draft_papers/question_drafts/draft_assets/review_events` + 索引 | 130 题写入为 130 文档；审核操作产生 `review_events` |
| **P1-B** | MinerU Worker（GPU锁/重试/取消/去重/清缓存） | 任务队列 + 单 GPU 串行 | 25 页 PDF OOM 后自动重试成功；重复 PDF 命中去重不再烧显存 |
| **P2-A** | AI 异步审核闭环 | `ai_review_tasks/ai_review_results` | 提交后浏览器不等待，结果回填为"建议"，人工确认 |
| **P2-B** | 三栏审核界面 | 左PDF/中OCR/右编辑器 + 过滤/快捷键 | 逐题审核效率显著提升，发布前预检 422 |

**本轮建议起手：P0-A + P0-B**（先不改页面），这是让"存入草稿箱 → 发布进题库"真正打通、且数据可追溯的最小闭环。

## 11. 数据迁移与兼容

- **旧草稿**（单 doc 整卷 `package`）：写一次性迁移脚本，按 `package.questions` 拆成一题一档，`review`/`edits` map 下沉到各题 `review_status`；原 doc 归档不删。
- **发布端零改动**：`questions` 正式库继续同时保留 V2 blocks + 兼容旧练习页的扁平字段（`material/explanation` 由 group/solution 投影），小程序端无需变更。
- **灰度**：新链路先跑通一套真题端到端，验收后再批量迁移历史 OCR job。

## 12. 风险与回滚

| 风险 | 缓解 |
|---|---|
| 一题一档后写入放大（130 次 add） | 批量并发写（沿用 `saveMany` concurrency=10）；导入任务化后可后台执行 |
| module_id 常量不一致 | 落地前统一到单一常量表，切题/前端/发布共用 |
| Worker 单 GPU 成为吞吐瓶颈 | 可接受（8G 显存现实约束）；后续可加"降分辨率/分段"提速 |
| 迁移出错 | 旧数据只归档不删除，可回滚到旧 `question_drafts` 单 doc 模式 |
